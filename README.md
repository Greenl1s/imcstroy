# Учёт приборов

Веб-приложение для учёта инструментов и оборудования: выдача, возврат, передача,
бронирование, поверки, календарь, история операций.

Работает целиком на своём сервере. Внешние сервисы не нужны.

## Как устроено

```
web/          Фронтенд: HTML, CSS, ванильный JavaScript (ES-модули)
server/       API: Node.js + Express
db/           Схема базы данных (выполняется автоматически при первом запуске)
scripts/      Перенос данных из старой версии на Supabase
```

Три контейнера: **Caddy** (сайт + HTTPS) → **API** (проверка прав) → **PostgreSQL**
(данные). База наружу не смотрит.

Развёртывание описано в [DEPLOY.md](DEPLOY.md).

## Возможности

* Роли: администратор и пользователь.
* Приборы: добавление, редактирование, удаление, списание, восстановление.
* Операции: взять, вернуть, передать другому, забронировать, подтвердить бронь.
* Календарь: поверки, выдачи и брони с цветовой индикацией.
* История: полный журнал операций по каждому прибору.
* Фильтры по состоянию, поверке и пользователю; поиск.
* Массовое списание и удаление.
* Фотографии приборов, QR-коды, тёмная тема, работа с телефона.

## Модель данных

| Таблица | Что хранит |
|---|---|
| `users` | Логин, **хэш** пароля, роль, контакты |
| `instruments` | Прибор и его текущее состояние: `free` / `busy` / `booked` / `retired` |
| `instrument_photos` | Фотографии, по одной на прибор |
| `history` | Журнал событий. Только добавление: записи не меняются и не удаляются |

Ключевые решения:

* **Списание — это статус, а не отдельная таблица.** Прибор не переезжает
  никуда и не меняет свой номер.
* **Состояние прибора проверяет база.** Прибор не может числиться свободным
  и одновременно быть у кого-то на руках — такую строку Postgres просто
  не запишет.
* **Все переходы состояния защищены от гонок.** Если двое одновременно нажмут
  «Взять», прибор достанется одному, а второй увидит понятную ошибку.
* **Пароли хранятся только в виде хэша** (bcrypt) и не покидают сервер.
* **Роль проверяется на сервере** при каждом запросе, а не в браузере.

Полная схема с комментариями: [`db/001_schema.sql`](db/001_schema.sql).

## Что изменилось по сравнению с версией на Supabase

| Было | Стало |
|---|---|
| Браузер ходил в базу напрямую с ключом доступа | Браузер общается только с API |
| Роль `admin` проверялась в JavaScript — обходилась через консоль | Роль в подписанном токене, проверяется сервером |
| Пароли лежали открытым текстом и читались любым посетителем | Хэши bcrypt, наружу не отдаются |
| Удаление и списание жили только в памяти браузера и откатывались после F5 | Изменения выполняет база |
| История удалялась целиком и перезаписывалась при каждом действии | Журнал только пополняется |
| Каждое действие выгружало все данные обратно (сотни запросов) | Один запрос на одно действие |
| Ручные номера приборов, у списанных — с приписанным нулём | Номера генерирует база; инвентарный номер — отдельное поле |
| Все фотографии грузились при каждом входе | Фото загружается только при открытии карточки |

## Разработка

```bash
cd server
npm install
DATABASE_URL=postgres://pribory:пароль@localhost:5432/pribory \
JWT_SECRET=$(openssl rand -base64 48) \
ADMIN_PASSWORD=admin12345 \
npm run dev
```

Фронтенд — статические файлы, сборка не нужна. Достаточно отдать папку `web/`
любым локальным сервером и указать в `index.html` адрес API.
